# Screen Studio 컨셉 정합성 감사 — 우리 UX와 구조 비교

- 이슈: #103 (맵: #97)
- 감사일: 2026-07-11
- 기준 자료: Screen Studio 구조 조사(#98, `docs/research/screen-studio-structure.md`, PR #107) · 확정 스펙(맵 #52 Decisions: #54, #57, #58, #60, #61, #64) · 실제 코드(`src/main/`, `src/renderer/src/`, `CONTEXT.md`)
- 분류 기준: **정합**(Screen Studio 컨셉과 같은 방향으로 구현됨) / **의도된 차별점**(경량 편집 헌장·맵 확정 결정에 근거한 의도적 차이) / **갭**(컨셉 또는 우리 자신의 확정 스펙과 어긋나는 미해소 지점, 심각도 high/medium/low)

## 요약

구현된 표면(캡처 툴바·오버레이·에디터·라이브러리·트레이)은 Screen Studio 컨셉 및 확정 스펙과 대체로 정합한다. 가장 큰 갭은 **구 단일 창 시절의 shell/런처 창이 부팅 경로에 그대로 살아 있는 것**(high) — "창 없는 캡처" 패러다임을 우리 스스로 확정(#54, #64)하고도 앱을 켜면 여전히 런처 창이 뜬다. 그 외 갭은 Dock 정책 미완(medium), 마이크로모션 부재(medium), 커맨드 메뉴·렌더 파이프라인·타임라인 인터랙션 밀도(low)다. 카메라·마이크·컷 편집 등은 헌장상 범위 밖이므로 갭이 아닌 의도된 차별점으로 처리했다.

## 갭 목록 (심각도순)

### G1 — 레거시 shell/런처 창이 부팅·에러 경로에 잔존 (high)

- 증거: `src/main/index.ts` 부팅 시 `if (onboarded) createShellWindow()`, 캡처 error 상태에서 `showLauncher()`, `app.on('activate')`의 `ensureShellWindow()`. 렌더러 `App.tsx`의 shell 분기가 `IdleView`(드롭다운 대상 픽커 + 최근 목록)·`RecordingView`를 여전히 라우팅.
- 왜 갭인가: #54는 창 5종(툴바·오버레이·에디터·Welcome·라이브러리)으로 확정하며 preview/런처를 제거했고, #64는 "녹화 정지→에디터 자동 생성"으로 상태를 재배선했다. 현 코드는 새 표면과 구 표면이 공존한다 — 앱을 켜면 Screen Studio식 '창 없는' 첫인상 대신 구 런처가 뜬다. 진입 표면 이중화는 컨셉 정합성의 핵심 훼손.
- 방향: 부팅=트레이 상주만(창 없음), error 상태는 툴바/알림 표면으로 이관, `IdleView`·`RecordingView`·shell role 제거.

### G2 — Dock 숨김(accessory) 정책 미적용 (medium)

- 증거: `app.dock.hide()/show()`가 shell 창 표시 여부에 종속(`index.ts`). `setActivationPolicy('accessory')`/LSUIElement 부재.
- 왜 갭인가: #54 확정은 "메뉴바 전용·Dock 숨김". 현재는 shell이 뜨는 동안 Dock 아이콘이 보인다. G1 해소와 한 몸.

### G3 — 마이크로모션·reduced-motion 부재 (medium)

- 증거: `index.css`에 transition/animation 6곳, `@keyframes` 1개(pulse), `prefers-reduced-motion` 미대응.
- 왜 갭인가: Screen Studio는 spring 마이크로 피드백(타임라인 페이드·wiggle)+reducedMotion 대응이 UI 품질의 축. 우리 C-dark(#56)는 "발랄한 다크"를 표방하나 모션 규칙은 미수립 상태 — 발랄함이 색·형태에만 있고 움직임에 없다. 접근성(reduced-motion)은 스펙과 무관하게 채워야 할 기본기.

### G4 — 에디터 커맨드 메뉴(⌘K) 부재 (low)

- Screen Studio의 키보드 생산성 장치. 우리 스펙에 미결정 항목이라 위반은 아니나, 에디터 단축키가 Space/←→/Esc뿐이라 조작 밀도 격차가 있다. v1 필수는 아님.

### G5 — 프리뷰 렌더 2D canvas (low)

- 증거: `EditorView` RAF 루프 + `drawComposition`(2D context). SS는 Pixi.js/WebGL.
- 고해상도(네이티브 2×) 소스에서 줌 애니메이션·스크럽의 부드러움 격차 가능성. 현 스코프(경량 편집)에선 병목 보고 없음 — 성능 문제가 관측될 때 승격.

### G6 — 타임라인 인터랙션 밀도 (low)

- SS: 스크러버 드래그 속도 비례 재생, 핀치 줌, 편집 wiggle 피드백. 우리: 클릭 시킹·드래그 이동/리사이즈·트림 핸들까지. 오디오 스크럽은 범위 밖(오디오 자체가 out of scope). 줌/스크럽 고급 인터랙션은 fast-follow 후보.

- 액센트 그린·SF Pro Rounded·마스코트·radius 8/12/16 — #56 C-dark 확정(SS의 퍼플·Inter·10px 자리를 의도적으로 대체).

## 결론

정합 12 / 의도된 차별점 11 / 갭 6(high 1 · medium 2 · low 3). 갭의 실체는 "Screen Studio를 못 따라간 것"이 아니라 **우리가 확정한 토폴로지 전환(#54·#64)의 마지막 마일(shell 제거·Dock 정책)이 미완**인 것 하나(G1+G2)와, 모션 언어 미수립(G3) 하나로 수렴한다. G1·G2는 한 티켓으로 묶어 해소할 것을 권한다. G4~G6은 관측되는 불편이 생길 때 티켓화해도 늦지 않다.
