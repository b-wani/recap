# 온보딩 UX는 Cap을 클린룸 모방한다

최초 실행 온보딩(권한 확보 + 기능 소개)을 설계하면서, 레퍼런스가 필요했다. Screen Studio는 소스가 비공개라 동작을 추정할 수밖에 없고, Cap(CapSoftware/Cap)은 소스가 공개되어 정책·동작·유저 플로우를 정확히 검증할 수 있다. 그래서 Cap의 온보딩을 기준으로 삼되 **클린룸 방식**으로 모방한다: 정책·동작·플로우 구조(단계식 온보딩 + 완료 플래그, 권한 단계 활성 중 250ms 폴링, 화면 녹화·손쉬운 사용 둘 다 필수, 시스템 설정 패널 직접 열기, 재시작 확인 다이얼로그)만 따르고, 코드·문구·에셋은 전부 신규 작성한다.

## Considered Options

- **Cap 코드 직접 포팅 / UI 에셋·문구 복사** — Cap 데스크톱 앱 코드는 AGPLv3. 파생물이 되면 Recap 전체에 AGPLv3 공개 의무가 생길 수 있어 탈락. 동작·아이디어의 모방은 저작권 대상이 아니므로 클린룸이면 안전하다.
- **Screen Studio 관찰 기반 자체 설계** — 스크린샷으로 화면은 볼 수 있지만 폴링 주기·재시작 처리·엣지 케이스 같은 내부 동작은 추정에 의존하게 됨. 검증 가능한 레퍼런스(Cap)가 있는데 추정으로 설계할 이유가 없어 탈락.

## Consequences

- 손쉬운 사용 권한은 현재 키 입력 오버레이(v2)에만 쓰이지만, v2 착수가 임박해 온보딩에서 화면 녹화와 함께 필수로 선확보한다.
- Cap의 단계 중 Recap에 대응물이 없는 것(Instant/Screenshot 모드 소개, 마이크·카메라 권한, 설정 토글 단계)은 싣지 않는다. 기능 소개 콘텐츠는 Recap 것(녹화·자동 효과 / 경량 편집 / 익스포트)으로 치환한다.
- 온보딩 완료 후 권한이 회수되는 경우는 온보딩이 다시 뜨지 않고, 기존 반응형 안전망(녹화 시점 `ensureScreenAccess` + 오류 화면)이 처리한다 — Cap과 동일한 책임 분리.
- 향후 Cap 코드를 참고할 때도 같은 원칙을 지킨다: 동작은 읽어서 배우되, 코드를 번역·복사하지 않는다.
